# プルリクエスト作成ルール

`za:pr` が PR を作るときに従うルール。

## 言語

日本語で書く。コード識別子・コマンドは原語のまま。
Claude の署名（`Co-Authored-By` など）は **付けない**。

## base ブランチと作業ブランチ

- base ブランチは `main`。
- 作業ブランチは `feature/{issue番号}-{title}`（`za:fix-issue` が作る）。
- ブランチ名が `feature/*` でない場合は PR を作らず、その旨を伝えて終了する。
- ブランチ名 `feature/{issue_id}-{title}` から issue ID（数値）を抽出する。
  抽出できなくても処理は継続する。

## 作成の流れ

1. `git branch --show-current` で現在のブランチを確認する。
2. `feature/*` でなければ終了する。
3. ブランチ名から issue ID を抽出する（取得できなくても継続）。
4. `git log main..HEAD --oneline` でコミット履歴を確認する。
5. `git diff main...HEAD` で変更差分を確認する。
6. 下記テンプレートに沿って本文を作成する。
7. 未 push なら `git push` でブランチをリモートへ送る。
8. `gh pr create` で作成する。本文は **HEREDOC** で渡す。作成後、PR の URL を表示する。

## タイトル

コミット履歴から簡潔にまとめる。issue ID が取得できた場合は、タイトルの先頭に
`#{issue_id}` を付ける（例: `#42 設定画面を追加する`）。

## 本文テンプレート

`.github/pull_request_template.md` の見出しに沿って記述する。

```markdown
## 概要

## 対応Issue

## 変更内容

## テスト
```

各セクションの埋め方:

| セクション | 内容 |
|------------|------|
| **概要**     | コミット履歴と差分から変更内容を要約（1〜3 文）。 |
| **対応Issue** | issue ID が取れた場合は `- fixes: https://github.com/{owner}/{repo}/issues/{issue_id}` の形で箇条書き（マージで自動クローズ）。取れない場合は「対応issueなし」と記載。 |
| **変更内容** | コミット履歴と差分から主な変更点を箇条書き。 |
| **テスト**   | 追加・変更したテストコードの内容を箇条書き。変更がない場合は「なし」。 |

## 運用

- 実装途中でレビューだけ先にもらいたい場合は `--draft` で下書き PR にする。
- 同じブランチの PR が既にある場合は新規作成せず、既存 PR を更新する。
- PR を作る前に、本プロジェクト（Rust）のビルド・テストを通しておく:
  `cargo build` / `cargo clippy`（警告なし）/ `cargo fmt` / `cargo test`。
- レビュアー指定の運用があればここに追記する。

# 6. Preview generation stays on the UI thread (coalesced), not async

Date: 2026-06-03
Status: accepted

## Context

プレビューパネル（テキスト/画像/音声）は `FilerComponent::build_preview_panel` で同期的に構築する。n/p によるファイル切り替え（#211）を入れた結果、連打すると 1 キーごとにフルリビルドが走り UI がフリーズしうる（#282）。テキストは最大 1 万行読み込み、画像はフルデコード、音声はデバイスを開いてデコーダを生成する。

本プロジェクトの方針（CLAUDE.md / ADR 0001）は「長時間の FS 操作は Async Job としてワーカースレッドで実行」であり、素直に考えればプレビュー生成も非同期化したくなる。しかし**プレビューのコンポーネント型はスレッド間で送れない（Send 不可）**:

- **音声**: rodio の `Sink` / `OutputStream` はオーディオデバイス/スレッドに束縛され Send 不可。
- **画像**: ratatui_image の `Picker` / `StatefulProtocol` は UI スレッド固有（protocol は端末を問い合わせる Picker から生成）。
- **テキスト**のみ（`Vec<String>`）が Send 可能。

したがってワーカースレッドで音声・画像のプレビューパネルを構築して返すことはできず、テキストだけ非同期化する中途半端な構成にしかならない。

## Decision

- **プレビュー生成は UI スレッド上の同期処理のままとし、Async Job 機構には載せない。**
- 連打フリーズの緩和は**再生成の coalesce（間引き）**で行う: n/p は Cursor File を動かして dirty フラグを立てるだけ。App は最大 `PREVIEW_REBUILD_THROTTLE`（約 120ms）に 1 回、かつ入力がアイドルになったとき（`next_event` のタイムアウト）に必ず 1 回、パネルを再生成する。単発操作はスロットル窓を既に超えているため即時再生成され、連打/長押しは間引かれ、停止後は最終位置を確実に再生成する。
- 実装: `FilerComponent::move_preview_cursor`（移動のみ）+ `build_preview_panel`（再生成）、`App::flush_preview_rebuild`、`SidePanel::is_preview`。

## Considered options

- **Async Job でワーカースレッド化**: 却下。音声（Sink/OutputStream）・画像（Picker/StatefulProtocol）が Send 不可でパネルをスレッド外で構築・返送できない。テキストのみ可能で不整合かつ複雑、しかも最も重い音声/画像を逃がせないため効果が薄い。
- **チャネル drain 型 coalesce（try_recv で連続 n/p を吸う）**: 却下。フレーム間にチャネルへ洪水的に溜まるイベント（キー長押しのオートリピート）しか合成できず、手動の速いタップはフレームに分散して合成されない。さらにキーと FileChange が同一チャネルのため取り扱いが複雑。スロットルはタップ・長押し双方に効く。
- **デバウンスタイマー（最後の n/p から N ms 待つ）**: 却下。単発操作にも遅延が乗る。スロットル+アイドルフラッシュなら単発は即時。
- **何もしない（既存のサイズ上限に委ねる）**: 却下。連打時の 1 キーごとデコードでフリーズが残る。

## Consequences

- 巨大な単一ファイルへ n/p で 1 回移動した場合は、その 1 回のビルドで UI が一瞬ブロックする。サイズ上限（テキスト 100MB/1万行、画像 4096²px）で最悪ケースは抑制されており、これは許容する（#282 が対象とする連打ケースではなく、ユーザーが意図した 1 回の読み込み）。
- 連続した高速ナビゲーション中は、表示中のプレビューがカーソルから最大 ~120ms（1 スロットル窓）遅れることがあり、停止後 ~100ms 以内に最終ファイルへ追従する。これは意図した coalesce の挙動。
- 将来 Send 可能なプレビュー型（純粋にテキスト由来のビュー等）が出れば個別に非同期化の余地はあるが、現状は全プレビューの構築を UI スレッドで統一する。
- 非同期化を再検討する場合は、音声/画像型の Send 可能な代替か、「デコードはワーカー・構築は UI スレッド」の分割パイプラインが必要になる。

# 9. Text preview syntax highlighting via syntect (shared dark theme with markdown)

Date: 2026-06-06

Status: accepted

## Context

テキストプレビュー（`v` キー、`PreviewComponent` の通常テキスト経路）は内容を無色のプレーンテキストで表示していた。ソースコードやマークアップを閲覧する際にシンタックスハイライトがあると構造を把握しやすく、プレビューの実用性が上がる（#284）。

設計ツリーの分岐: ハイライト用ライブラリの選定、配色（テーマ・色モデル）、最大 1 万行のハイライト負荷の扱い、ファイル種別の判定範囲、既存コンポーネントへの組み込み方。

前提として、ADR 0008（マークダウンプレビュー）で基盤の大半が既に整っている:

- `syntect`（onig 込み）は `tui-markdown` の `highlight-code` 機能経由で**既に依存ツリーにあり**、native runner + glibc ビルド（ADR 0003）で問題なくリンクできている。
- `TextOutputState` は既に `Vec<Line<'static>>`（スタイル付き行）を保持でき、`render_text_output` も Span のスタイルを描画する。
- tui-markdown のコードブロックは `syntect` + テーマ `base16-ocean.dark` + 前景色のみ（`as_24_bit_terminal_escaped(_, false)`）で色付けしている。

したがって #284 の本質は「ファイル種別判定 → ハイライト → `Vec<Line<'static>>` 化 → 既存テキスト経路へ接続」に絞られる。

## Decision

- **`syntect` を直接依存に追加**する（既定 features、onig）。既にツリーにある同一バージョン（5.3.0）であり、onig も ADR 0008 で受容済みのため、依存・ビルドへの追加コストは実質ない。
- **配色は `base16-ocean.dark` の前景色のみ**を使う（背景は端末に委ねる）。tui-markdown のコードブロックと**同一の配色**になり、「md 内コードブロック」と「ソースファイル単体プレビュー」で同じ言語が同じ色になる。固定ダークテーマで端末テーマには追従しないが、これは ADR 0008 が既に選択・受容した妥協であり、それと足並みを揃える。
- **ハイライトはプレビュー構築時に全行（最大 1 万行）一括**で行う。ADR 0006 のとおりプレビュー構築は UI スレッド上の同期処理で coalesce 済みであり、「単一の重いファイルで一度だけ UI がブロックしうるが、サイズ上限（100MB / 1 万行）で最悪ケースは抑制される」方針に従う。描画は毎フレーム可視範囲のみ再構築する既存設計のため、ハイライトのコストは構築 1 回に集約され毎フレームには乗らない。
- **種別判定は拡張子 → ファイル名トークン（Makefile 等）→ 先頭行（shebang 等）の順**。`syntect` の `find_syntax_by_extension` / `find_syntax_by_first_line` を使う。特定できなければ `None` を返し、**プレーン表示にフォールバック**する。
- **`syntect` のスタイルを ratatui の `Span` へ直接変換**する（`ui::syntax`）。tui-markdown がやっている「ANSI 文字列へ変換 → `ansi-to-tui` で再パース」の往復は無駄なので避け、`highlight_line` の `(syntect::Style, &str)` から `Span`（`fg = Color::Rgb`、背景なし）を直接組む。`highlight_line` 失敗時はその行をプレーンにフォールバックする。
- **既存テキスト経路へ統合**する。`PreviewComponent::new` がテキスト読み込み後に `ui::syntax::to_lines` を試行し、`Some` なら `with_styled_lines`、`None` なら従来どおり `with_lines`。`build_preview_panel` の分岐や `SidePanel` バリアントは増やさず、n/p の coalesce（ADR 0006）もそのまま効く。マークダウン経路（`new_markdown`、レンダリング表示）とメッセージ表示（`with_message`）は不変。
- 読み込み上限・バイナリ判定・truncation は既存の `TextPreview` を流用する。

## Considered options

- **ANSI 16 色へマッピングして端末テーマに追従**: 却下。ライト/ダーク両対応にできるが、(1) 24bit 固定のマークダウンコードブロックと色が食い違う、(2) スコープ→16 色のマッピングを自前実装する必要がある、(3) 表現力が落ちる。配色一貫性を優先した。
- **軽量な独自簡易ハイライタ**: 却下。対応言語が限定的で保守負担が大きい。`syntect` が既にツリーにあり多言語対応（Rust 含む）のため不要。
- **可視範囲のみ遅延ハイライト**: 却下。`syntect` は行を先頭から逐次処理するため、可視窓だけ色付けするには parse 状態のキャッシュが要り実装が複雑化する。1 万行上限と coalesce で十分に抑制できるため、構築時一括で足りる。
- **`new_highlighted` を新設し `build_preview_panel` に分岐追加（markdown と同型）**: 却下。シンタックスハイライトはテキストプレビューの既定挙動であり別モードではないため、テキスト経路に畳み込む方が配線が少なく一貫する。
- **`tui-markdown` 同様に `ansi-to-tui` 経由で変換**: 却下。ANSI への往復が無駄。`syntect::Style` から `Span` を直接組めば済む。

## Consequences

- すべてのテキストプレビューが、種別を特定できれば色付き・できなければプレーンになる。マークダウンは従来どおりレンダリング表示で、ソースとしてのハイライトはしない。
- 配色は端末テーマに追従しない固定ダークテーマのため、ライト端末では `base16-ocean.dark` の前景色が見づらい場合がある。マークダウンコードブロックと同じ妥協であり、許容する。将来、端末テーマ追従やテーマ選択を入れる場合は、マークダウン側と合わせて見直す。
- 1 万行近い巨大ソースを開くと、その構築 1 回で UI が一瞬ブロックしうる（ADR 0006 が許容する単発の重い読み込みと同じ範囲）。問題が顕在化すれば、ハイライト専用の行数ガードを追加する余地がある。
- `syntect` のスコープ→色変換は前景色のみで、太字・斜体等のフォントスタイルは反映しない（tui-markdown のコードブロックと同じ挙動）。
- 対応言語は `syntect` の既定構文セットに依存する（Rust・TOML・JSON・YAML 等を含む）。バンドルにない言語はプレーン表示になる。
- マークダウンコードブロックとの「同一配色」は、`tui-markdown` の `highlight-code` 機能（既定有効・`base16-ocean.dark` 固定）に追従依存する。将来 `tui-markdown` を `default-features = false` 化したり、同クレートがテーマを変えたりすると、この一貫性は静かに崩れる。その際は本モジュールのテーマと合わせて見直す。

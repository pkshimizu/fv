# Issue 作成ルール・コミット規約

`za:issue` が issue を作るとき、`za:fix-issue` / `za:review` がコミットするときに従うルール。

## 言語

issue・コミットメッセージともに日本語で書く。コード識別子・コマンドは原語のまま。
Claude の署名（`Co-Authored-By` など）は **付けない**。

## Issue 作成ルール

### 分割の方針（INVEST）

要望を分析し、INVEST 原則に基づいて適切な粒度の issue に分割する。
要望が大きい場合は複数の issue に、小さい場合は 1 つの issue にまとめる。

- **I (Independent)**: 他の issue に依存せず独立して完了できる。
- **N (Negotiable)**: 実装方法に柔軟性がある。
- **V (Valuable)**: 単体で価値を提供する。
- **E (Estimatable)**: 作業量を見積もれる大きさである。
- **S (Small)**: 1 つが大きくなりすぎない。
- **T (Testable)**: 完了条件が明確でテスト可能である。

順序依存が避けられない場合は、本文の「依存」欄に先行 issue を明記する。

### テンプレートの選択

各 issue の分類に応じて、`.github/ISSUE_TEMPLATE/` から対応するテンプレートを選ぶ。
テンプレートの `labels:` がそのまま付与するラベルになる。

| テンプレート | 分類 | 用途 |
|--------------|------|------|
| `feat.yml`     | `feat`     | 新しい機能の実装 |
| `fix.yml`      | `fix`      | 不具合修正 |
| `docs.yml`     | `docs`     | ドキュメントの変更 |
| `style.yml`    | `style`    | コードフォーマットの修正 |
| `refactor.yml` | `refactor` | 挙動を変えないリファクタリング |
| `perf.yml`     | `perf`     | パフォーマンス・チューニング |
| `test.yml`     | `test`     | テストコードの追加・変更 |
| `chore.yml`    | `chore`    | ビルド・ツール・ライブラリの変更 |

### タイトル

`{分類}: {内容}` の形式（例: `feat: ユーザー認証機能の追加`）。
端的に、何をするかが分かるように。

### 本文

選んだテンプレートの見出しに合わせてマークダウンで記述する（例: `feat.yml` なら
概要・動機/背景・詳細・代替案）。テンプレートの required フィールドは必ず埋める。

za で扱う際に有用な以下の項目は、テンプレートの末尾に追記してよい。

```markdown
## 受け入れ条件
- [ ] {完了と判断できる条件}

## 依存
- {先に完了が必要な issue。無ければ「なし（すぐ着手可）」}

## 参照
- 元プラン: {docs/plans/ のファイルパス}
```

### 作成コマンド

ユーザーの承認を得たら、`gh` コマンドで作成する。分類とラベルを一致させる。

```bash
gh issue create --title "{分類}: {タイトル}" --label "{分類}" --body "{本文}"
```

複数作成する場合は順番に作成し、各 issue の URL を記録して報告する。

## コミット規約

`za:fix-issue` / `za:review` はこの規約でコミットする。

### メッセージ形式

```
{種別}: {要約}

{本文：何を・なぜ変えたか。必要なら}

{対応 issue があれば: 例) #42}
```

- **種別**: `feat` / `fix` / `docs` / `style` / `refactor` / `perf` / `test` / `chore`
  （issue の分類・ラベルと対応）。
- **要約**: 日本語で簡潔に。命令形で「〜する」調に統一。
- **issue 参照**: 対応する issue があれば本文に `#番号` を含める。「Closes #番号」は PR 側で使う。
- **粒度**: 1 コミット 1 論理変更。複数の関心事が混ざるなら分割する。
- Claude の署名（`Co-Authored-By` など）は付けない。

### 例

```
feat: 設定画面を追加する

メニューから開ける設定画面を追加。表示・保存の基本動作を実装する。

#12
```

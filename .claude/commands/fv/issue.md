GitHub Issueを作成してください。

ユーザーの要望: $ARGUMENTS

## 手順

### 1. 要望の分析

ユーザーの要望を分析し、INVEST原則に基づいて適切な粒度のissueに分割してください。

- **I (Independent)**: 各issueが他のissueに依存せず独立して完了できること
- **N (Negotiable)**: 実装方法に柔軟性があること
- **V (Valuable)**: 各issueが単体で価値を提供すること
- **E (Estimatable)**: 作業量が見積もり可能な大きさであること
- **S (Small)**: 1つのissueが大きくなりすぎないこと
- **T (Testable)**: 完了条件が明確でテスト可能であること

要望が大きい場合は複数のissueに分割してください。小さい場合は1つのissueで構いません。

### 2. テンプレートの選択

各issueの内容に応じて、`.github/ISSUE_TEMPLATE/` から適切なテンプレートを選択してください。

- `feat.yml`: 新しい機能の実装
- `fix.yml`: 不具合修正
- `docs.yml`: ドキュメントの変更
- `style.yml`: コードフォーマットの修正
- `refactor.yml`: リファクタリング
- `perf.yml`: パフォーマンス・チューニング
- `test.yml`: テストコードの追加・変更
- `chore.yml`: ビルドプロセス、ツール、ライブラリの変更

### 3. ユーザーへの確認

issueを作成する前に、以下の内容をユーザーに提示して確認を取ってください。

- 作成予定のissue一覧（タイトル、分類、概要）
- 分割の妥当性
- 不明点や追加で確認が必要な詳細

### 4. issueの作成

ユーザーの承認を得たら、MCP経由（`gh` コマンド）でissueを作成してください。

issueのタイトルは `<分類>: <内容>` の形式にしてください（例: `feat: ユーザー認証機能の追加`）。

テンプレートの各フィールドに対応する内容を本文に含めてください。本文はマークダウン形式で、テンプレートの見出しに合わせて記述してください。

```bash
gh issue create --title "<分類>: <タイトル>" --label "<分類>" --body "<本文>"
```

複数のissueを作成する場合は、順番に作成し、各issueのURLを記録してください。

### 5. 結果の報告

作成したすべてのissueのURLをユーザーに報告してください。
